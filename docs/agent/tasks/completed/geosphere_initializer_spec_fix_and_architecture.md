## Part 1 ‚Äî Immediate Spec Fix (do this now)

### Problem
`spec/services/terra_sim/geosphere_initializer_spec.rb:158` and `:174` are
failing because `allow_any_instance_of` in the `before` block tries to stub
`determine_regolith_depth` and `determine_particle_size` on
`TerraSim::GeosphereInitializer` ‚Äî but those methods don't exist yet. The stub
fires before the `skip` guard in the `it` block can run, causing a hard failure
instead of a clean skip.

### Fix
In the `before` block at approximately line 150-156, add a `method_defined?`
check alongside the existing `column_exists?` check:

```ruby
before do
  earth_atmosphere
  if ActiveRecord::Base.connection.column_exists?(:geospheres, :regolith_depth) &&
     TerraSim::GeosphereInitializer.method_defined?(:determine_regolith_depth)
    allow_any_instance_of(TerraSim::GeosphereInitializer).to receive(:determine_regolith_depth).and_return(3.0)
    allow_any_instance_of(TerraSim::GeosphereInitializer).to receive(:determine_particle_size).and_return(0.5)
  end
end
```

## Part 2 ‚Äî Architecture Backlog (do NOT implement yet, StarSim is WIP)

### Context
During investigation of the spec failures, a full architectural review of
`GeosphereInitializer` and `SystemBuilderService` was conducted. The following
decisions were made and must be preserved for when StarSim procedural
generation is built out.

### Current State (as of March 15, 2026)
- `TerraSim::GeosphereInitializer` ‚Äî **dead code, never called anywhere**
- `StarSim::PlanetarySeedGenerator` ‚Äî **dead code, never called anywhere**
- `StarSim::SystemBuilderService#create_geosphere` ‚Äî correctly reads
  `geosphere_attributes` directly from JSON and calls `build_geosphere`. This
  is the working path for all seeded systems (Sol, Alpha Centauri, AOL-732356).
- `seeds.rb` ‚Äî calls `SystemBuilderService` only. No direct geosphere seeding.

### The Correct Architecture (for when StarSim procedural path is built)

**Two distinct paths:**

1. **Seeded systems** (Sol, Alpha Centauri, any hand-authored JSON)
   ‚Üí `SystemBuilderService` ‚Üí reads `geosphere_attributes` from JSON directly
   ‚Üí `build_geosphere` with real data
   ‚Üí Already works correctly. Do not change.

2. **Procedural systems** (generated by `PlanetarySeedGenerator`)
   ‚Üí `GeosphereInitializer` is the intended service for this path
   ‚Üí Currently broken ‚Äî uses hardcoded `case @body_type` switches that ignore
     physical parameters entirely
   ‚Üí Not wired up to anything yet

### What GeosphereInitializer Must Become

When the procedural path is built, `GeosphereInitializer` must be refactored
to compute geosphere properties from physical parameters, not type switches:

- `geological_activity` ‚Äî derive from mass, density, age, orbital zone
- `tectonic_activity` ‚Äî derive from mass and internal heat retention
- `crust_composition` ‚Äî derive from orbital zone (inner = silicate/metal,
  outer = ices), stellar type, and mass. Do NOT hardcode per body_type.
- `regolith_depth` ‚Äî compute from geological_activity, surface gravity,
  presence/absence of atmosphere, and body age
- `regolith_particle_size` ‚Äî compute from weathering mechanisms available
  (thermal cycling, micrometeorite flux, wind if atmosphere present)
- `weathering_rate` ‚Äî already exists in geosphere model, use it

**Resolution order for any field:**
1. JSON-specified (known/observed data) ‚Äî always wins
2. Computed from physical parameters (inferred)
3. `nil` ‚Äî explicitly unknown, do not fabricate

### Data Confidence
No `data_confidence` field exists on geospheres yet. When procedural
generation is built, add a simple enum column:
- `observed` ‚Äî mission data, direct measurement
- `modeled` ‚Äî scientifically constrained inference
- `generated` ‚Äî procedural computation from orbital/stellar parameters

This is geosphere-level only (not per-field) to keep schema lean.

### Known/Unknown Data Philosophy
The game reflects reality: many geosphere properties are unknown or assumed.
- Sol bodies: use mission data where it exists (Mars rover regolith,
  MESSENGER Mercury data, Cassini Titan data)
- Alpha Centauri: only confirmed observational data ‚Äî Proxima b has mass and
  orbital period, no geosphere data. Leave absent, do not fabricate.
- Procedural bodies: compute what physics constrains, nil what it doesn't.
- `sol-complete.json` already has correct `geosphere_attributes` per body.
  Luna, Mars, Mercury have real regolith data that should be added to the JSON.

### What NOT to Do
- Do not call `GeosphereInitializer` from `SystemBuilderService` ‚Äî seeded
  systems already work correctly without it
- Do not add `core_materials`/`mantle_materials`/`crust_materials` config
  tables ‚Äî composition comes from JSON for known bodies, physics for procedural
- Do not randomize geological_activity with `rand()` ‚Äî use physical derivation

### Files to Modify When StarSim is Ready
- `app/services/terra_sim/geosphere_initializer.rb` ‚Äî full refactor
- `app/services/star_sim/planetary_seed_generator.rb` ‚Äî wire to initializer
- `db/migrate/` ‚Äî add `data_confidence` enum to geospheres
- `app/data/solar_systems/sol-complete.json` ‚Äî add regolith_attributes for
  Luna, Mars, Mercury from real mission data
