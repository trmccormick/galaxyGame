# Star System Naming Standards

## Overview
This document outlines the naming conventions for star systems, stars, and planets in the Galaxy Game. The naming system distinguishes between scientific catalog identifiers (used during exploration) and settlement names (assigned upon colonization).

## Scientific Catalog Identifiers
- **Format**: `[Letters]-[Numbers]` (e.g., "AOL-732356")
- **Usage**: Used during exploration and initial system analysis
- **Generation**: Randomly generated by `NameGeneratorService#generate_identifier`
- **Transition**: Converted to settlement names upon first habitation module landing

## Settlement Names

### Star System Names
- **Source**: Drawn from `config/names.yml` star_systems section
- **Categories**: Star Trek, Star Wars, Babylon 5, Battlestar Galactica, Dune
- **Fallback**: Generic names like "New Eden", "Haven", "Providence"
- **Uniqueness**: Ensured through prefix/suffix addition if conflicts exist

### Star Names
- **Format**: `[System Name]-[Letter]` (e.g., "Gaia-A")
- **Primary Star**: Always suffixed with "-A"
- **Multiple Stars**: "-B", "-C", etc. for binary/trinary systems

### Planet Names
- **Primary Habitable Planet**: `[System Name] Prime` (e.g., "Gaia Prime")
- **Secondary Planets**: `[System Name] [Roman Numeral]` for sequential naming (e.g., "Gaia II", "Gaia III")
- **Unique Names**: Thematic names for planets with special characteristics (e.g., "Topaz", "Aurora", "Spire")
- **Moon Naming**: `[Planet Name] [Roman Numeral]` (e.g., "Gaia Prime I", "Topaz II")

## Implementation Details

### NameGeneratorService Methods
- `generate_identifier()`: Creates scientific catalog IDs
- `generate_star_name(system_identifier, index)`: Creates star identifiers (e.g., "AOL-732356-A")
- `generate_planet_name(star_name, planet_index)`: Creates planet names using Prime/Roman numeral system
- `generate_settlement_star_name()`: Selects settlement names from YAML configuration

### YAML Configuration
Located at `config/names.yml`:
- `star_systems`: Categorized by sci-fi universe with planet counts and notable worlds
- `planet_names.generic`: Fallback settlement names

## Current System Examples

### Eden System (AOL-732356)
- **System**: Eden (settlement name)
- **Star**: Eden-A
- **Planets**: Mix of thematic names with type-appropriate suffixes:
  - Eden Prime (primary habitable - terrestrial)
  - Gaia Minor (secondary habitable - terrestrial)
  - Aurora (unique thematic - terrestrial)
  - Butte (geological formation - rocky)
  - Spire (geological formation - rocky)
- **Moons**: Roman numerals (e.g., "Eden Prime I", "Aurora II")

## Rules and Guidelines

1. **Prime Designation**: Typically reserved for primary habitable planets, but can be used thematically for planet types
2. **Consistent Theming**: Each planet type can use different naming conventions consistently:
   - **Terrestrial**: Mix of thematic names with "Prime"/"Minor" suffixes
   - **Oceanic**: Greek letters (Alpha, Beta, Gamma, etc.) for water-themed worlds
   - **Rocky**: Greek letters for geological formations
   - **Other types**: Thematic suffixes appropriate to their classification
3. **Greek Letter Usage**: When used, follow proper alphabetical order:
   - Alpha (Α/α), Beta (Β/β), Gamma (Γ/γ), Delta (Δ/δ), Epsilon (Ε/ε)
   - And continuing through the Greek alphabet
4. **Roman Numerals**: Used for moons and sequential planet naming within systems
5. **Unique Names**: Thematic names should be unique within reasonable scope
6. **Thematic Consistency**: Planet names should fit their world's characteristics
6. **Moon Naming**: Always use Roman numerals for moons

## JSON Import Compatibility

### Aliases Field Issue
**Status**: ✅ FIXED - Aliases moved to properties field  
**Problem**: Adding `aliases` field to JSON star system data caused import failures.

**Solution Implemented**:
- Moved `aliases` from top-level JSON field into `properties` JSONB field
- `SystemBuilderService` properly imports aliases via properties merging
- Maintains backward compatibility for name resolution

**Current JSON Structure**:
```json
{
  "name": "Eden II", 
  "properties": {
    "aliases": ["Topaz", "Eden II"]
  }
}
```

**Impact**: Aliases now persist through system regeneration, enabling reliable name resolution for renamed celestial bodies.
